# EMAPA source notes

EMAPA (Mouse Developmental Anatomy Ontology, <https://obofoundry.org/ontology/emapa.html>) is used
in Babel as an anatomy source.

In this integration, Babel:

- extracts EMAPA identifiers for anatomy processing;
- exports EMAPA xref concords from UberGraph;
- includes EMAPA in anatomy compendium merge inputs.

Related implementation files:

- `src/createcompendia/anatomy.py`
- `src/snakefiles/anatomy.snakefile`
- `config.yaml`

Additional source details are documented in:

- `download.md` — how EMAPA data is retrieved (via the UberGraph SPARQL endpoint) and what
  queries are used.
- `filtering.md` — which root term the identifier extraction starts from and how the
  `part_of` traversal works.
- `mappings.md` — how EMAPA cross-references to other prefixes are extracted from UberGraph
  xrefs and written as concord rows.
- `impact-report.md` — auto-generated report quantifying the identifiers, biolink types, and
  clique changes EMAPA adds to the anatomy compendium. See "Adding a new data source" in
  `CLAUDE.md` for how to regenerate it.
- `clique-diff.md` — a full build-vs-build diff of the anatomy compendia with and without EMAPA,
  covering the restructured and dropped cliques the impact report cannot see. Confirms that
  adding EMAPA changes no pre-existing clique at all: every difference is a new EMAPA-only clique.
- `multi-emapa-xrefs.csv` — every UBERON term that cross-references more than one EMAPA term, one
  row per mapping, regenerated by `scripts/multi_emapa_uberon_xrefs.py`. This is the evidence
  behind the `anatomy_unique_prefixes` decision below.

## Two decisions that need SME confirmation

### EMAPA is not a unique prefix (SME check)

`EMAPA` is deliberately **not** listed in `anatomy_unique_prefixes` (`config.yaml`), so a single
clique may hold several EMAPA identifiers.

122 UBERON terms cross-reference more than one EMAPA term (263 EMAPA CURIEs in total); for 116 of
them, more than one of those targets is a live, labelled EMAPA term. Reading the list, these are
overwhelmingly **genuine 1:n mappings** rather than mapping errors — a mouse developmental
partonomy naming a structure at finer or stage-specific granularity than the adult UBERON term:

| UBERON term | EMAPA terms it xrefs |
|---|---|
| [`UBERON:0000044`](http://purl.obolibrary.org/obo/UBERON_0000044) "dorsal root ganglion" | `EMAPA:16667` "spinal ganglion", `EMAPA:16668` "dorsal root ganglion", `EMAPA:18372` "tail dorsal root ganglion" |
| [`UBERON:0001017`](http://purl.obolibrary.org/obo/UBERON_0001017) "central nervous system" | `EMAPA:16470` "central nervous system", `EMAPA:16754` "tail central nervous system" |
| [`UBERON:0001049`](http://purl.obolibrary.org/obo/UBERON_0001049) "neural tube" | `EMAPA:16164` "neural tube", `EMAPA:16757` "tail neural tube" |
| [`UBERON:0001021`](http://purl.obolibrary.org/obo/UBERON_0001021) "nerve" | `EMAPA:32808` "nerve", `EMAPA:32815` "peripheral nerve" |
| [`UBERON:0001074`](http://purl.obolibrary.org/obo/UBERON_0001074) "pericardial cavity" | `EMAPA:16131` "pericardial component", `EMAPA:16132` "pericardial cavity" |

The full list is in [`multi-emapa-xrefs.csv`](./multi-emapa-xrefs.csv).

Restricting `EMAPA` would block all 122 merges, and — because `glom()` keeps whichever competing
CURIE it sees first and a loser with no ids-file row is dropped outright — it withdrew three
already-published identifiers from the compendia. Leaving EMAPA unrestricted keeps each UBERON
term's EMAPA mappings together and makes this PR purely additive (see
[clique-diff.md](./clique-diff.md)).

**What an SME should confirm:** that collapsing e.g. "spinal ganglion", "dorsal root ganglion" and
"tail dorsal root ganglion" into one clique with `UBERON:0000044` is acceptable. The alternative —
restricting EMAPA — does not produce a *better* clique; it produces an arbitrary winner among the
three plus two singletons, and can silently delete identifiers. If these mappings should not be
treated as equivalences at all, the right fix is to filter the offending UBERON xrefs (see
`docs/sources/CLAUDE.md`, "An OBO `hasDbXref` is not an equivalence"), not to restrict the prefix.

### EMAPA terms are not tagged with a taxon (SME check)

We considered attaching the taxon
[`NCBITaxon:10090`](http://purl.obolibrary.org/obo/NCBITaxon_10090) "Mus musculus" to every EMAPA
term, since EMAPA is by definition a mouse ontology. **We decided against it.**

The tempting precedent is MP, where tagging the taxon cleanly separates MP (mammal) terms from HP
(human) ones. EMAPA is not analogous, because EMAPA terms do not all sit in EMAPA-only cliques. Of
the 8,096 EMAPA CURIEs in the built compendia, **4,343 share a clique with a UBERON term** and the
remaining 3,753 form new EMAPA-only cliques (see [clique-diff.md](./clique-diff.md)). UBERON is a
multi-species ontology covering a broad range of animals, so marking that first group as
mouse-specific would be **incorrect** —
[`UBERON:0001049`](http://purl.obolibrary.org/obo/UBERON_0001049) "neural tube" is not a mouse
structure just because `EMAPA:16164` is one of its cross-references.

A taxon attaches to the *clique*, not to the EMAPA identifier within it, so there is no way to say
"mouse, but only for the EMAPA member". Tagging only the EMAPA-only cliques would be defensible on
its own terms but would make the taxon a function of whether UBERON happens to xref a term, which
is not a meaningful distinction to expose to consumers.

**What an SME should confirm:** that no taxon is preferable to a mouse taxon here. If per-identifier
(rather than per-clique) taxon information is ever wanted, that is a different feature and should be
designed as one.
